### 1. 数据库存储（审查历史）

#### 功能说明
将每次代码审查的结果存储到数据库中，支持查询历史记录、统计趋势、生成报告。

#### 为什么有用

| 场景 | 现有方案 | 有数据库后 |
|------|---------|----------|
| 查看某 PR 的历史审查 | 无记录 | ✅ 可查询 |
| 统计哪类问题最多 | 无数据 | ✅ 趋势分析 |
| 审查超时/失败 | 丢失 | ✅ 可重试 |
| 团队代码质量趋势 | 靠感觉 | ✅ 量化指标 |
| CI/CD 集成 | 无 | ✅ 可校验 |

#### 实现方案
```python
# 数据库模型
class ReviewRecord:
    id: int
    pr_id: int
    repo_owner: str
    repo_name: str
    status: str  # success, failed, partial
    comments_count: int
    critical_count: int
    created_at: datetime
    completed_at: datetime

class ReviewComment:
    id: int
    review_id: int
    file: str
    line: int
    severity: str  # critical, high, medium, low
    category: str  # security, bug, style
    message: str
```

### 3. Webhook 签名验证

#### 功能说明
启用 GitHub Webhook 的 HMAC-SHA256 签名验证，防止伪造请求攻击。

#### 为什么有用

| 风险 | 无验证 | 有验证后 |
|------|--------|----------|
| 伪造请求 | 可被利用 | ✅ 无法伪造 |
| DDoS 攻击 | 直接打到业务 | ✅ 可识别来源 |
| 数据泄露 | 可能 | ✅ 较安全 |

#### 安全风险示例
```python
# 如果不验证签名，攻击者可以：
# 1. 模拟 GitHub Webhook 请求
# 2. 触发任意 PR 的代码审查
# 3. 消耗 API 调用配额
# 4. 在你的服务上执行自定义代码（如果有注入漏洞）
```

#### 实现方案
```python
# 取消注释 webhook.py 中的验证代码
def verify_webhook_signature(payload: bytes, signature: str) -> bool:
    expected = hmac.new(
        settings.github_webhook_secret.encode(),
        payload,
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(f"sha256={expected}", signature)
```

#### 优先级评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 业务价值 | ⭐⭐⭐⭐⭐ | 安全是底线 |
| 实现复杂度 | ⭐☆☆☆☆ | 已有代码，只需启用 |
| 依赖关系 | 无 | 独立功能 |
| **综合优先级** | **5/5** | **必须实现** |

#### 预估工作量
- 取消注释：10 分钟
- 测试验证：1 小时
- 文档更新：30 分钟
- **总计：约 2 小时**

### 6. C++ 静态分析集成

#### 功能说明
集成 C++ 静态分析工具（clang-tidy, cppcheck），对 C++ 代码进行确定性检测。

#### 为什么有用

| 场景 | 纯 LLM | LLM + 静态分析 |
|------|--------|---------------|
| C++ 内存泄漏 | 靠猜测 | ✅ 确定性检测 |
| 性能问题 | 定性 | ✅ 定量分析 |
| 规则覆盖 | 不稳定 | ✅ 全面 |
| 误报率 | 较高 | ✅ 可控 |

#### 工具对比

| 工具 | 优点 | 缺点 |
|------|------|------|
| clang-tidy | LLVM 生态，准确定位 | 仅 C++ |
| cppcheck | 支持多语言，规则丰富 | 误报较多 |
| Coverity | 商业级，最准确 | 收费昂贵 |

#### 实现方案
```python
class CppAnalyzer:
    def analyze(self, filepath: str, code: str) -> list[Comment]:
        # 1. 保存临时文件
        with tempfile.NamedTemporaryFile(suffix=".cpp") as f:
            f.write(code)
            f.flush()

            # 2. 运行 clang-tidy
            result = subprocess.run(
                ["clang-tidy", f.name],
                capture_output=True,
                text=True
            )

            # 3. 解析输出
            return self._parse_output(result.stdout)

        # 4. 运行 cppcheck
        result = subprocess.run(
            ["cppcheck", "--enable=all", f.name],
            capture_output=True,
            text=True
        )
```

#### 优先级评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 业务价值 | ⭐⭐⭐⭐⭐ | 显著提升准确性 |
| 实现复杂度 | ⭐⭐⭐⭐☆ | 需要编译环境 |
| 依赖关系 | 无 | 独立功能 |
| **综合优先级** | **3/5** | **C++ 项目必须** |

#### 预估工作量

### 7. 消息队列（Redis/Kafka）

#### 功能说明
将同步请求改为异步处理，通过消息队列解耦。

#### 为什么有用（详细分析）

| 方面 | 同步现状 | 异步后 |
|------|---------|--------|
| Webhook 响应时间 | 5-30 秒 | < 100ms |
| 并发处理 | 无 | 多 Worker |
| 失败恢复 | 丢失 | 重试队列 |
| GitHub 限流 | 难以控制 | 可控消费速率 |
| 突发流量 | 积压/崩溃 | 队列缓冲 |

#### 性能对比

```
场景：同时收到 10 个 PR 事件

同步模式:
┌────┐
│ PR1│ ───→ [████████████████████] 10秒 ───→ 完成
│ PR2│ ───→ [等待中...]                 ───→ 10秒后开始
│ ...│ ───→ [等待中...]                      ...
└────┘
总耗时：100秒+

异步模式（Redis + 3 Worker）:
┌────┐
│ PR1│ ──→ [Queue] ──→ [Worker1] ──→ 10秒 ──→ 完成
│ PR2│ ──→ [Queue] ──→ [Worker2] ──→ 10秒 ──→ 完成
│ PR3│ ──→ [Queue] ──→ [Worker3] ──→ 10秒 ──→ 完成
└────┘
总耗时：~40秒（3个并行）
```

#### 优先级评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 业务价值 | ⭐⭐⭐☆☆ | 小规模不明显 |
| 实现复杂度 | ⭐⭐⭐☆☆ | 需要运维 |
| 依赖关系 | 无 | 独立功能 |
| **综合优先级** | **2/5** | **大流量时才需要** |

#### 预估工作量
- Redis 集成：2 天
- Worker 实现：2 天
- 监控/告警：1 天
- 部署文档：1 天
- **总计：约 6 天**
